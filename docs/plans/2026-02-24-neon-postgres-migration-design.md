# Neon PostgreSQL Migration Design

**Date:** 2026-02-24
**Status:** Approved
**Replaces:** MotherDuck (cloud DuckDB) as production database

## Problem

MotherDuck's "Remote catalog has changed" error occurs on every write operation from the Shiny app deployed on Posit Connect. The error is an architectural limitation of MotherDuck's ATTACH-based catalog sync protocol — not fixable with retry logic, connection management, or DuckDB configuration. See conversation context for full root cause analysis.

## Decision

Migrate from DuckDB/MotherDuck to Neon PostgreSQL (serverless Postgres) as the single database for all environments: local development, Posit Connect production, Python sync scripts, and GitHub Actions.

## Neon Configuration

- **Provider:** AWS
- **Region:** us-east-2 (Ohio)
- **Postgres version:** 17
- **Database:** neondb
- **Connection pooling:** Built-in PgBouncer (pooler endpoint)
- **Auth:** Disabled (app has its own admin_users + bcrypt auth)

## Architecture

### Before

```
Local Dev (Windows) → DuckDB file (data/local.duckdb)
Posit Connect (Linux) → MotherDuck (cloud DuckDB via ATTACH)
Python sync scripts → DuckDB / MotherDuck (dual-mode)
```

### After

```
Local Dev (Windows) → Neon PostgreSQL
Posit Connect (Linux) → Neon PostgreSQL
Python sync scripts → Neon PostgreSQL
GitHub Actions → Neon PostgreSQL
```

## R Connection Layer

Replace `connect_db()` / `connect_motherduck()` / `connect_local()` with a `pool` object:

```r
library(pool)
library(RPostgres)

db_pool <- dbPool(
  RPostgres::Postgres(),
  host = Sys.getenv("NEON_HOST"),
  dbname = Sys.getenv("NEON_DATABASE", "neondb"),
  user = Sys.getenv("NEON_USER"),
  password = Sys.getenv("NEON_PASSWORD"),
  port = 5432,
  sslmode = "require",
  minSize = 1,
  maxSize = 5
)
```

### What changes

- **No more `rv$db_con`** — pass `db_pool` directly to `dbGetQuery()`/`dbExecute()`. Pool handles connection lifecycle.
- **No more `safe_query()`/`safe_execute()`** — retry/reconnect logic unnecessary. Pool auto-validates connections. Keep tryCatch + notify for user-facing error messages.
- **No more `next_id()` helper** — use `INSERT ... RETURNING column_id` with auto-increment.
- **No more `connect_db()` / `disconnect()`** — pool manages everything. Cleanup via `onStop(function() poolClose(db_pool))`.

## Schema Changes

### Auto-increment IDs

All `INTEGER PRIMARY KEY` become `INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`.

`GENERATED BY DEFAULT` (not `GENERATED ALWAYS`) allows explicit ID inserts from Python sync scripts.

### Restore FK Constraints

| Column | References | On Delete |
|--------|-----------|-----------|
| `results.tournament_id` | `tournaments(tournament_id)` | CASCADE |
| `results.player_id` | `players(player_id)` | RESTRICT |
| `results.archetype_id` | `deck_archetypes(archetype_id)` | SET NULL |
| `results.pending_deck_request_id` | `deck_requests(request_id)` | SET NULL |
| `tournaments.store_id` | `stores(store_id)` | RESTRICT |

### SQL Dialect Changes

| DuckDB Pattern | Postgres Equivalent |
|---------------|---------------------|
| `sprintf()` bulk INSERT VALUES | Same (or `dbWriteTable()` for bulk) |
| `LOWER(x) LIKE LOWER(?)` | `x ILIKE ?` |
| `next_id()` MAX+1 | `INSERT ... RETURNING id` |
| `INSTALL/LOAD motherduck` | Removed |
| `ATTACH 'md:...'` | Removed |
| `BOOLEAN` literals | Same (TRUE/FALSE) |
| `CURRENT_TIMESTAMP` | Same |
| `CREATE OR REPLACE VIEW` | Same |
| `DECIMAL(9,6)` | Same |

### Timestamps

DuckDB `TIMESTAMP DEFAULT CURRENT_TIMESTAMP` works identically in Postgres. No changes needed.

## Write Operations Migration

~99 write operations across 10 R files:

1. Replace `rv$db_con` with `db_pool` in all `dbGetQuery()`/`dbExecute()` calls
2. Remove `safe_query()` wrapper — replace with `dbGetQuery(db_pool, ...)` inside tryCatch
3. Remove `safe_execute()` wrapper — replace with `dbExecute(db_pool, ...)` inside tryCatch
4. Remove `next_id()` calls — use `RETURNING` clause
5. Keep soft-delete patterns (good practice) but hard deletes now work via FK cascades

## Data Migration

One-time process:

1. Export all 18 tables from local DuckDB to CSV
2. Create Postgres schema on Neon (new `db/schema.sql`)
3. Import CSVs into Neon (respecting FK order: scenes first, then stores, players, etc.)
4. Verify row counts match
5. Reset identity sequences to `MAX(id) + 1` for each table

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| RPostgres doesn't support `channel_binding=require` | Use `sslmode=require` only (sufficient for security) |
| Bulk INSERT via sprintf in ratings.R | Test with Postgres; fall back to `dbWriteTable()` if needed |
| Identity sequence drift after CSV import | Script to reset sequences after migration |
| Neon cold start latency (~500ms) | First request slightly slow; pool keeps connection warm after |
| Free tier limits (0.5GB, 100 compute-hours) | Current data is small; monitor usage in dashboard |
