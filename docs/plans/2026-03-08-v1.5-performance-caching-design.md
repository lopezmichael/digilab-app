# v1.5.0 — Performance & Caching Design

**Date:** 2026-03-08
**Status:** Planning
**Roadmap IDs:** PERF1–PERF6

---

## Goal

Reduce initial page load time and improve responsiveness across all tabs by deferring unnecessary work, caching expensive computations, and tuning database queries.

---

## Current State

### What's already in place
- **28 `bindCache()` calls** across public outputs (dashboard 16, players 2, meta 1, tournaments 1, stores 5, shared 1)
- **`safe_query()`/`safe_execute()`** wrappers with retry logic for prepared statement collisions
- **Connection pool:** min 1 / max 5 connections via `pool` package
- **`player_ratings_cache`** table pre-computed on startup
- **Admin lazy loading** — admin modules only sourced after login
- **45+ database indexes** covering most WHERE/JOIN patterns
- **~331 total queries**, all parameterized

### What's missing
- All 6 public server modules load eagerly — every tab's reactives fire on first page load
- No materialized views — dashboard aggregations computed live on every cache miss
- Uncached outputs exist (modals, some mobile views)
- No query timing instrumentation
- Pool settings (min 1, max 5) never tuned against real usage
- Startup queries run individually, not batched

## Phase 1: Instrument & Measure (PERF1)

### 1A. Query timing in safe_query()

Add timing instrumentation to `safe_query()` and `safe_execute()` in `shared-server.R`. Log queries exceeding a threshold (e.g., 200ms) with the query text and execution time.

```r
safe_query <- function(pool, query, params = NULL, default = data.frame()) {
  start <- proc.time()["elapsed"]
  # ... existing logic ...
  elapsed <- proc.time()["elapsed"] - start
  if (elapsed > 0.2) {
    message(sprintf("[SLOW QUERY %.1fms] %s", elapsed * 1000, substr(query, 1, 120)))
  }
  result
}
```

## Phase 2: Optimize Queries & Caching (PERF2, PERF3, PERF5, PERF6)

### 2A. Materialized views (PERF2)

Pre-compute expensive JOIN + GROUP BY aggregations into flat tables. The app then filters at read time using simple WHERE clauses — no joins, no aggregation, no prepared statement collision risk.

**Design principle: per-store grain.** All views store data at the `store_id` level (the most granular useful level). This enables all current and future filter patterns:

| Filter level | SQL on materialized view |
|-------------|--------------------------|
| Scene (current) | `WHERE scene_id = $1` |
| Country (future) | `WHERE country = 'Brazil'` |
| State (future) | `WHERE state_region = 'Texas'` |
| Single store (future) | `WHERE store_id = $1` |
| Online (current) | `WHERE is_online = TRUE` |
| Global | No WHERE (or no scene filter) |

The app aggregates up from per-store rows at query time, which is trivially fast on a flat table.

**Existing regular views to replace:** `player_standings`, `archetype_meta`, `store_activity` (defined in `db/schema.sql:465-536`) are currently unused by the app but define similar aggregations. These will be dropped and replaced by the materialized views below.

#### View 1: `mv_player_store_stats`

Pre-computes per-player-per-store stats. Replaces the 4-table JOIN in Players tab (~lines 138-154 in public-players-server.R) and the main_decks CTE (~lines 157-173).

```sql
CREATE MATERIALIZED VIEW mv_player_store_stats AS
WITH player_deck_counts AS (
  SELECT r.player_id, t.store_id, da.archetype_name, da.primary_color,
         COUNT(*) as times_played,
         ROW_NUMBER() OVER (
           PARTITION BY r.player_id, t.store_id
           ORDER BY COUNT(*) DESC
         ) as rn
  FROM results r
  JOIN tournaments t ON r.tournament_id = t.tournament_id
  JOIN deck_archetypes da ON r.archetype_id = da.archetype_id
  WHERE da.archetype_name != 'UNKNOWN'
  GROUP BY r.player_id, t.store_id, da.archetype_name, da.primary_color
)
SELECT
  p.player_id,
  p.display_name,
  s.store_id,
  s.scene_id,
  sc.country,
  sc.state_region,
  s.is_online,
  t.format,
  COUNT(DISTINCT r.tournament_id) as events,
  SUM(r.wins) as wins,
  SUM(r.losses) as losses,
  SUM(r.ties) as ties,
  ROUND(SUM(r.wins) * 100.0 / NULLIF(SUM(r.wins) + SUM(r.losses), 0), 1) as win_pct,
  COUNT(CASE WHEN r.placement = 1 THEN 1 END) as firsts,
  COUNT(CASE WHEN r.placement <= 3 THEN 1 END) as top3s,
  pdc.archetype_name as main_deck,
  pdc.primary_color as main_deck_color
FROM players p
JOIN results r ON p.player_id = r.player_id
JOIN tournaments t ON r.tournament_id = t.tournament_id
JOIN stores s ON t.store_id = s.store_id
JOIN scenes sc ON s.scene_id = sc.scene_id
LEFT JOIN player_deck_counts pdc
  ON p.player_id = pdc.player_id AND s.store_id = pdc.store_id AND pdc.rn = 1
GROUP BY p.player_id, p.display_name, s.store_id, s.scene_id,
         sc.country, sc.state_region, s.is_online, t.format,
         pdc.archetype_name, pdc.primary_color;

CREATE UNIQUE INDEX ON mv_player_store_stats
  (player_id, store_id, format);
```

**App query (scene leaderboard):**
```sql
SELECT player_id, display_name,
       SUM(events) as "Events", SUM(wins) as "W", SUM(losses) as "L", SUM(ties) as "T",
       ROUND(SUM(wins) * 100.0 / NULLIF(SUM(wins) + SUM(losses), 0), 1) as "Win %",
       SUM(firsts) as "1sts", SUM(top3s) as "Top 3s"
FROM mv_player_store_stats
WHERE scene_id = $1 AND format = $2
GROUP BY player_id, display_name
HAVING SUM(events) >= $3
```

#### View 2: `mv_archetype_store_stats`

Pre-computes per-archetype-per-store stats. Replaces Meta tab query (~line 65-79), dashboard deck analytics (~line 308-323), and meta share timeline (~line 513-527).

```sql
CREATE MATERIALIZED VIEW mv_archetype_store_stats AS
SELECT
  da.archetype_id,
  da.archetype_name,
  da.primary_color,
  da.secondary_color,
  da.display_card_id,
  da.is_multi_color,
  s.store_id,
  s.scene_id,
  sc.country,
  sc.state_region,
  s.is_online,
  t.format,
  date_trunc('week', t.event_date)::date as week_start,
  COUNT(r.result_id) as entries,
  COUNT(CASE WHEN r.placement = 1 THEN 1 END) as firsts,
  COUNT(CASE WHEN r.placement <= 3 THEN 1 END) as top3s,
  SUM(r.wins) as total_wins,
  SUM(r.losses) as total_losses,
  COUNT(DISTINCT r.player_id) as pilots,
  COUNT(DISTINCT r.tournament_id) as tournaments
FROM deck_archetypes da
JOIN results r ON da.archetype_id = r.archetype_id
JOIN tournaments t ON r.tournament_id = t.tournament_id
JOIN stores s ON t.store_id = s.store_id
JOIN scenes sc ON s.scene_id = sc.scene_id
WHERE da.is_active = TRUE AND da.archetype_name != 'UNKNOWN'
GROUP BY da.archetype_id, da.archetype_name, da.primary_color, da.secondary_color,
         da.display_card_id, da.is_multi_color,
         s.store_id, s.scene_id, sc.country, sc.state_region, s.is_online,
         t.format, date_trunc('week', t.event_date);

CREATE UNIQUE INDEX ON mv_archetype_store_stats
  (archetype_id, store_id, format, week_start);
```

**App query (meta tab):**
```sql
SELECT archetype_id, archetype_name as "Deck", primary_color as "Color",
       SUM(entries) as "Entries",
       SUM(firsts) as "1sts",
       SUM(top3s) as "Top 3s",
       ROUND(SUM(total_wins) * 100.0 / NULLIF(SUM(total_wins) + SUM(total_losses), 0), 1) as "Win %"
FROM mv_archetype_store_stats
WHERE scene_id = $1 AND format = $2
GROUP BY archetype_id, archetype_name, primary_color
HAVING SUM(entries) >= $3
ORDER BY SUM(entries) DESC
```

**App query (meta share timeline — replaces 4-table JOIN in dashboard):**
```sql
SELECT week_start, archetype_name, primary_color, SUM(entries) as entries
FROM mv_archetype_store_stats
WHERE scene_id = $1
GROUP BY week_start, archetype_name, primary_color
ORDER BY week_start, entries DESC
```

#### View 3: `mv_tournament_list`

Pre-computes tournament listing with winner info. Replaces the LATERAL JOIN in Tournaments tab (~line 63-80) and Dashboard recent tournaments.

```sql
CREATE MATERIALIZED VIEW mv_tournament_list AS
SELECT
  t.tournament_id,
  t.event_date,
  t.event_type,
  t.format,
  t.player_count,
  t.rounds,
  s.store_id,
  s.name as store_name,
  s.slug as store_slug,
  s.scene_id,
  sc.country,
  sc.state_region,
  s.is_online,
  p.display_name as winner_name,
  da.archetype_name as winning_deck
FROM tournaments t
JOIN stores s ON t.store_id = s.store_id
JOIN scenes sc ON s.scene_id = sc.scene_id
LEFT JOIN LATERAL (
  SELECT r2.player_id, r2.archetype_id
  FROM results r2
  WHERE r2.tournament_id = t.tournament_id AND r2.placement = 1
  ORDER BY r2.result_id LIMIT 1
) r ON true
LEFT JOIN players p ON r.player_id = p.player_id
LEFT JOIN deck_archetypes da ON r.archetype_id = da.archetype_id;

CREATE UNIQUE INDEX ON mv_tournament_list (tournament_id);
```

**App query (tournaments tab):**
```sql
SELECT tournament_id, event_date as "Date", store_name as "Store",
       event_type as "Type", format as "Format", player_count as "Players",
       rounds as "Rounds", winner_name as "Winner", winning_deck as "Winning Deck"
FROM mv_tournament_list
WHERE scene_id = $1 AND format = $2
ORDER BY event_date DESC
```

#### View 4: `mv_store_summary`

Pre-computes store stats. Replaces store listing queries (~line 949-966 in public-stores-server.R) and dashboard store count.

```sql
CREATE MATERIALIZED VIEW mv_store_summary AS
SELECT
  s.store_id,
  s.name,
  s.slug,
  s.address,
  s.city,
  s.state,
  s.zip_code,
  s.latitude,
  s.longitude,
  s.website,
  s.country,
  s.is_online,
  s.scene_id,
  sc.country as scene_country,
  sc.state_region,
  COUNT(DISTINCT t.tournament_id) as tournament_count,
  COUNT(DISTINCT r.player_id) as unique_players,
  COALESCE(ROUND(AVG(t.player_count), 1), 0) as avg_players,
  MAX(t.event_date) as last_event
FROM stores s
JOIN scenes sc ON s.scene_id = sc.scene_id
LEFT JOIN tournaments t ON s.store_id = t.store_id
LEFT JOIN results r ON t.tournament_id = r.tournament_id
WHERE s.is_active = TRUE
GROUP BY s.store_id, s.name, s.slug, s.address, s.city, s.state, s.zip_code,
         s.latitude, s.longitude, s.website, s.country, s.is_online,
         s.scene_id, sc.country, sc.state_region;

CREATE UNIQUE INDEX ON mv_store_summary (store_id);
```

#### View 5: `mv_dashboard_counts`

Pre-computes the dashboard value box metrics per scene/format combo. Replaces the core metrics query (~line 335+ in public-dashboard-server.R).

```sql
CREATE MATERIALIZED VIEW mv_dashboard_counts AS
SELECT
  s.scene_id,
  sc.country,
  sc.state_region,
  s.is_online,
  t.format,
  t.event_type,
  COUNT(DISTINCT t.tournament_id) as tournament_count,
  COUNT(DISTINCT r.player_id) as player_count,
  COUNT(DISTINCT s.store_id) as store_count,
  COUNT(DISTINCT r.archetype_id) as deck_count,
  COUNT(r.result_id) as total_entries
FROM tournaments t
JOIN stores s ON t.store_id = s.store_id
JOIN scenes sc ON s.scene_id = sc.scene_id
LEFT JOIN results r ON t.tournament_id = r.tournament_id
LEFT JOIN deck_archetypes da ON r.archetype_id = da.archetype_id
  AND da.archetype_name != 'UNKNOWN'
GROUP BY s.scene_id, sc.country, sc.state_region, s.is_online,
         t.format, t.event_type;

CREATE UNIQUE INDEX ON mv_dashboard_counts
  (scene_id, COALESCE(format, ''), COALESCE(event_type, ''), is_online);
```

#### Refresh strategy

**When:** After any data mutation — result submission, tournament creation/edit, Limitless sync.

**How:** Single R helper function called wherever `rv$data_refresh` is incremented:

```r
refresh_materialized_views <- function(pool) {
  con <- pool::localCheckout(pool)
  views <- c("mv_player_store_stats", "mv_archetype_store_stats",
             "mv_tournament_list", "mv_store_summary", "mv_dashboard_counts")
  for (v in views) {
    tryCatch(
      DBI::dbExecute(con, sprintf("REFRESH MATERIALIZED VIEW CONCURRENTLY %s", v)),
      error = function(e) message(sprintf("[MV REFRESH ERROR] %s: %s", v, e$message))
    )
  }
}
```

**Cost:** At 600 results / ~100 tournaments, refresh of all 5 views takes <1 second total. Even at 10,000 results it would be a few seconds — well within acceptable for an admin action.

**Trigger points in code:**
- `admin-results-server.R` — after result submission
- `admin-tournaments-server.R` — after tournament create/edit/delete
- `scripts/sync_limitless.py` — after sync completes (via SQL or a post-sync R script)

### 2B. Expand bindCache coverage (PERF3)

**Update:** Modal `bindCache()` was attempted and reverted. `bindCache()` on `renderUI` that calls `showModal()` is fundamentally incompatible — cache hits return the HTML but skip the `showModal()` side effect, so the modal won't re-open after closing. All 4 detail modals (player, store, deck, tournament) must NOT use `bindCache()`.

Remaining candidates for `bindCache()`:

| Output | File | Cache keys |
|--------|------|------------|
| Mobile meta cards | `public-meta-server.R` | Same keys as desktop equivalent |
| Mobile tournament cards | `public-tournaments-server.R` | Same keys as desktop equivalent |
| Dashboard context text | `public-dashboard-server.R` | `dashboard_format`, `dashboard_event_type` |

**Note:** With materialized views (PERF2), most queries become fast enough that `bindCache()` provides diminishing returns. Focus caching on outputs with expensive R-side rendering (e.g., building many HTML card elements).

### 2C. Pool tuning (PERF5)

Review Neon connection pool settings:

| Setting | Current | Consider |
|---------|---------|----------|
| `minSize` | 1 | Keep at 1 (Neon charges for idle connections) |
| `maxSize` | 5 | Evaluate — may need increase if concurrent users grow |
| `idleTimeout` | default (60s) | Set explicitly to 600s to reduce reconnection overhead |
| `validationInterval` | default (600s) | Set explicitly to match Neon's idle timeout |

**Neon consideration:** Neon auto-suspends after 5 minutes of inactivity. Pool `validationInterval` should account for this — connections may go stale if the app is idle.

## Phase 3: Lazy Tab Loading (PERF4)

### Current behavior
All 6 public server modules are sourced at startup (`app.R:1067-1072`). Shiny computes reactive outputs as soon as their UI elements exist, meaning every tab fires queries on first page load.

### Proposed approach: Gate reactives on tab visibility

Add a tab-visited tracking mechanism:

```r
# In app.R — track which tabs have been visited
rv$visited_tabs <- reactiveVal(character(0))

# In shared-server.R — mark tab as visited on navigation
observeEvent(rv$current_nav, {
  visited <- rv$visited_tabs()
  if (!rv$current_nav %in% visited) {
    rv$visited_tabs(c(visited, rv$current_nav))
  }
})
```

Then in each server module, gate the main data reactive:

```r
# In public-meta-server.R
meta_archetype_data <- reactive({
  req(rv$current_nav == "meta" || "meta" %in% rv$visited_tabs())
  # ... existing query logic ...
})
```

**Tradeoff:** First visit to a tab will have a brief loading delay. Mitigated by skeleton loaders already in place.

### Alternative: Deferred module sourcing

Source public server modules lazily, similar to admin modules:

```r
# In app.R
observe({
  req(rv$current_nav == "meta")
  if (!meta_loaded()) {
    source("server/public-meta-server.R", local = TRUE)
    meta_loaded(TRUE)
  }
})
```

**Pros:** No queries fire at all until tab is visited.
**Cons:** More complex state management, potential flash of empty UI.

**Recommendation:** Start with the reactive gating approach (simpler, less risk). Move to deferred sourcing only if gating doesn't reduce startup load enough.
