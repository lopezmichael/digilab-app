# v1.4 Admin Improvements Design Document

**Created:** 2026-03-06
**Status:** Draft
**Target Release:** v1.4.0

## Overview

v1.4 focuses on reducing manual admin work, improving data quality through required fields,
and creating a unified request/notification system. The release improves all admin tabs
except Results and Upload (deferred to a paired redesign in a future version).

Key themes:
- **Request queue**: Persist all community requests to DB with approve/reject workflow
- **Notification widget**: Surface pending items to admins without tab-switching
- **iframe storage fix**: Resolve mobile localStorage bug via postMessage bridge
- **Announcement system**: Version modals and custom announcements for all users
- **Admin UX polish**: Search bars, filters, required fields, auto-messages

## 1. Infrastructure

### 1a. postMessage Iframe Storage Bridge

**Problem:** The Shiny app runs inside an iframe on `app.digilab.cards` (Vercel) but is
hosted on the Posit Connect Cloud domain. Mobile Safari and Chrome block or partition
third-party localStorage in cross-origin iframes. This causes:
- Welcome modal always showing on mobile (reported bug)
- Any future localStorage features (announcements, version modal) will also break

**Solution:** Use `window.postMessage()` to delegate storage to the parent frame.

**Flow:**
1. Shiny iframe sends: `parent.postMessage({ type: 'digilab-storage-get', key: 'X' }, '*')`
2. Parent page (`app.astro` on Vercel) reads from its own first-party localStorage
3. Parent responds: `iframe.contentWindow.postMessage({ type: 'digilab-storage-result', key: 'X', value: '...' }, '*')`
4. Writes work the same way: iframe requests, parent executes

**Files to modify:**
- `www/scene-selector.js` — Replace direct `localStorage` calls with postMessage wrappers
- `digilab-web/src/pages/app.astro` — Add message listener for storage relay

**Fallback:** If no parent frame (direct Posit Connect access), fall back to direct
localStorage (works fine in non-iframe context).

### 1b. `admin_requests` Table

Unified table for all community requests submitted via webhook modals.

```sql
CREATE TABLE admin_requests (
  id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  request_type TEXT NOT NULL,        -- 'store_request', 'scene_request', 'data_error', 'bug_report'
  scene_id INTEGER REFERENCES scenes(scene_id),
  status TEXT NOT NULL DEFAULT 'pending',  -- pending, approved, rejected, resolved
  payload JSONB NOT NULL,            -- flexible per request type
  discord_username TEXT NOT NULL,
  submitted_at TIMESTAMPTZ DEFAULT NOW(),
  resolved_at TIMESTAMPTZ,
  resolved_by TEXT,
  notes TEXT
);

CREATE INDEX idx_admin_requests_status ON admin_requests(status);
CREATE INDEX idx_admin_requests_type ON admin_requests(request_type);
CREATE INDEX idx_admin_requests_scene ON admin_requests(scene_id);
```

Store request:

## 2. Notification Widget

### Design

A horizontal notification bar displayed below the admin header, visible only to admins.

```
+--------------------------------------------------------------------+
| [bell icon] 3 deck requests - 2 store requests - 1 bug report     |
+--------------------------------------------------------------------+
```

Each item count is a clickable link navigating to the relevant admin tab/section.

### Behavior

- **Scene admins** see counts for their scene only (store requests, data errors, bug reports)
- **Super admins** see all counts, including deck requests and scene requests (super-admin-only tabs)
- Queries `admin_requests WHERE status = 'pending'` + `deck_requests WHERE status = 'pending'`
- Refreshes on admin tab load and when `rv$refresh_requests` invalidates
- Background refresh via `reactiveTimer(300000)` (5 minutes)
- Dismissible per-session (hides bar until new items appear)
- Hidden when zero pending items

### Implementation

```r
# In admin UI area
uiOutput("admin_notification_bar")

# In server
output$admin_notification_bar <- renderUI({
  # Invalidate on timer + manual refresh
  rv$refresh_requests
  admin_refresh_timer()

  counts <- get_pending_request_counts(db_pool, rv$current_scene, rv$is_superadmin)

  if (sum(unlist(counts)) == 0) return(NULL)

  # Build notification items as clickable links
  items <- list()
  if (counts$deck_requests > 0 && rv$is_superadmin) {
    items <- c(items, list(actionLink("notif_decks", paste(counts$deck_requests, "deck requests"))))
  }
  if (counts$store_requests > 0) {
    items <- c(items, list(actionLink("notif_stores", paste(counts$store_requests, "store requests"))))
  }
  # ... etc

  div(class = "admin-notification-bar", ...)
})
```

### CSS

```css
.admin-notification-bar {
  background: rgba(var(--bs-warning-rgb), 0.15);
  border-left: 3px solid var(--bs-warning);
  padding: 8px 16px;
  margin-bottom: 12px;
  border-radius: 4px;
  display: flex;
  align-items: center;
  gap: 8px;
}
```

## 3. Webhook Modal Improvements

### Current State

Three modals in `shared-server.R`:
- `show_store_request_modal()` — scene dropdown, store name, city/state, Discord (optional for existing scenes)
- `show_data_error_modal()` — description required, Discord optional
- `show_bug_report_modal()` — title + description required, Discord optional

### Changes

**All modals:**
- Make Discord username **required** on all modals (needed for follow-up)
- Disable submit button until required fields filled (`shinyjs::toggleState`)
- Persist to `admin_requests` table in addition to Discord webhook
- Show success message with "We'll follow up on Discord" text

**Store request modal** (separate modal from scene request):
- Pre-selects current scene, requires store name, city, state/country, Discord
- Link to switch to scene request modal if scene doesn't exist
- **Future:** Fuzzy store name matching — as user types, query existing stores in the
  selected scene and show "Did you mean?" suggestions to prevent duplicates

**Scene request modal** (separate modal):
- Requires city/area name, state/country, Discord username
- Optional: known stores (textarea, one per line), additional notes
- Stored as `request_type = 'scene_request'`, surfaces on Scenes tab for super admins
- **Future:** Fuzzy scene matching — as user types city name, show existing scenes
  that might match to prevent duplicate requests

**Data error modal:**
- Auto-attach current context (player name, tournament, store — whatever page they're on)
- Keep description required

**Bug report modal:**
- Keep title + description required

### DB Persistence Flow

```
User submits modal
  -> Validate required fields
  -> INSERT INTO admin_requests (type, scene_id, payload, discord_username)
  -> Fire Discord webhook (fire-and-forget, existing pattern)
  -> Invalidate rv$refresh_requests (updates notification widget)
  -> Show success toast
```

### Admin Resolution Flow & Discord Notifications

When an admin approves or rejects a request, the following happens:

```
Admin clicks Approve/Reject on request card
  -> UPDATE admin_requests SET status, resolved_at, resolved_by, notes
  -> Invalidate rv$refresh_requests (updates notification widget)
  -> If applicable: auto-post Discord webhook (see below)
  -> Show success toast
```

**Which resolutions post to Discord:**

| Action | #scene-requests (forum) | #scene-coordination (forum) | #scene-updates (public) |
|--------|------------------------|----------------------------|----------------------|
| New scene request submitted | Creates thread with request details | -- | -- |
| New scene created (approved) | -- | Creates thread with welcome message | Short announcement |
| Store added to existing scene | -- | Post in thread: "New store added: [Name]" | Short announcement |
| Deck request approved/rejected | -- | -- | -- |
| Data error resolved | -- | -- | -- |
| Bug report resolved | -- | -- | -- |

**Discord channel roles:**

| Channel | Category | Type | Purpose | Who posts |
|---------|----------|------|---------|-----------|
| `#scene-requests` | SCENE MANAGEMENT | Forum | Public intake for new scene requests. Community can see and comment on requests. Also serves as super admin alert to check the app. | App webhook + manual users |
| `#scene-coordination` | SCENE MANAGEMENT | Forum | Per-scene admin threads for onboarding and ongoing coordination. Created automatically when a scene is approved. Tags: continent of origin (North America, Europe, etc.) | App webhook + admins |
| `#scene-updates` | COMMUNITY | Text | Public announcements for new scenes and new stores. Short, clean posts only. | App webhook |

**Discord server structure (for reference):**
```
INFO:             #welcome, #announcements, #rules
COMMUNITY:        #general, #scene-updates
SUPPORT:          #bug-reports (forum), #feature-requests (forum), #data-corrections (forum), #resources
SCENE MANAGEMENT: #scene-requests (forum), #scene-coordination (forum)
OPS:              #error-log, #dev-log
```

**Request submission flow:**
- App-generated requests create a new forum thread in `#scene-requests` via webhook
  (`thread_name` parameter) with the request details (city, country, stores, requester)
- Manual requests: users can still post directly in `#scene-requests` without the app;
  super admin handles those manually or asks the user to submit through the app for tracking

**Approval flow:**
- New scene threads in `#scene-coordination` created via webhook (see Section 10)
- Store additions post to existing scene threads via `discord_post_to_scene()`

Individual-relevant resolutions (rejections, error fixes, bug fixes) are handled
manually — admin follows up via Discord DM as needed. No automated DM templates
for v1.4.

## 4. Welcome & Announcement Flow

### Modal Priority (one per page load)

1. **Welcome/Onboarding modal** — first-time visitors only
2. **Announcement modal** — returning visitors with unseen announcements
3. **Version changelog modal** — returning visitors after app update (lowest priority)

### Flow Logic

```
On page load:
  1. Parent frame sends stored values via postMessage
  2. Check digilab_onboarding_complete
     - If FALSE/missing -> show Welcome modal, stop
     - If TRUE -> continue
  3. Check digilab_last_seen_announcement_id vs latest active announcement
     - If newer announcement exists -> show Announcement modal, stop
     - Otherwise -> continue
  4. Check digilab_last_seen_version vs current app version
     - If different -> show Version modal
     - Otherwise -> no modal
```

**"Create Scene + Stores":**

1. Pre-fills the scene form only
2. Admin manually adds stores later on the Stores tab
3. Marks request as resolved

**"Reject":** Opens small confirmation with optional reason field.

### New Scene Welcome Message & Discord Thread Creation

The final step of scene onboarding is posting the welcome message to Discord. The app
**creates a new forum thread** in `#scene-coordination` via Discord's webhook API, which
supports the `thread_name` parameter for forum channels. This replaces the manual process
of creating threads and copy-pasting messages.

**Flow:**
1. Admin completes scene setup (scene created, admin assigned, stores added)
2. Admin clicks **"Post Welcome to Discord"** button on the Scenes tab
3. **Preview modal** opens showing the full message in an editable textarea
4. Admin reviews, optionally edits the message, then clicks **"Post"**
5. App sends webhook POST with `thread_name: "[Scene Name]"` to `#scene-coordination`
6. Discord creates a new forum thread with the welcome message as the first post
7. Discord returns the thread/message ID in the webhook response
8. App **automatically saves the thread ID** to `scenes.discord_thread_id` — no manual
   copy-paste needed. Future posts (new store notifications, data errors, etc.) route
   to this thread automatically.
9. Simultaneously posts a short announcement to `#scene-updates` (see below)
10. Button changes to "Thread Created" (disabled) to prevent double-posting

**Fallback:** If the webhook succeeds but the response can't be parsed (timeout, network
issue), show a warning: "Thread created but ID couldn't be saved automatically. Add it
manually in the Discord Thread ID field." Falls back to current manual process.

**Existing scenes:** Scenes with manually-entered `discord_thread_id` values are unaffected.
The automation only applies to new scenes created through the v1.4 flow.

### Discord Channels for Scene Events

Scene-relevant events post to **two** Discord channels:

| Event | #scene-coordination (forum) | #scene-updates (public) |
|-------|-----------------------------|----------------------|
| New scene created | New thread with full welcome message | Short announcement |
| New store added to scene | Post in existing thread: "New store added: [Name]" | Short announcement |

**#scene-coordination** is a forum channel. Each scene gets its own thread, created
automatically by the webhook when the welcome message is posted. The thread ID is stored
on the scene record for future posts (e.g., new store notifications).

**#scene-updates** is a public announcement channel. Short, clean announcements only:
```
New scene: [Scene Name] is now live on DigiLab! Check out the local
leaderboard and tournament history at https://app.digilab.cards
```

### Full Scene Onboarding Flow

End-to-end flow from request to live scene:

```
1. User submits "new scene needed" via store request modal in app
   -> Persists to admin_requests table (request_type = 'scene_request')
   -> Webhook creates new thread in #scene-requests forum with details
   -> Community can see and comment on the request thread

2. Super admin sees request on Scenes tab notification bar
   -> Opens request card, reviews details + any Discord thread discussion

3. Admin clicks "Create Scene + Stores" (or "Just Create Scene")
   -> Scene form pre-filled, admin reviews and saves
   -> Store stubs auto-created if applicable
   -> Request marked as resolved

4. Admin creates scene admin account on Users tab
   -> Copies welcome DM with credentials via "Copy Welcome DM" button
   -> Sends DM to new admin on Discord

5. Admin clicks "Post Welcome to Discord" on Scenes tab
   -> Preview modal with editable welcome message
   -> Posts to #scene-coordination (creates new forum thread)
   -> Optionally posts short announcement to #scene-updates
   -> Thread ID saved on scene record

6. Scene is live, admin is onboarded, community is informed
```

`#scene-requests` remains a permanent public-facing intake channel. Users who find the
Discord first can also post requests there manually without using the app.

Add **store count** and **admin count** columns to the scenes reactable.
Helps identify empty scenes (no stores) and uncovered scenes (no admin).

## Resolved Decisions

1. **Announcement admin UI**: Sub-section on the Scenes tab (global settings tab).
2. **Biweekly schedule**: Option B — "next occurrence" date picker to anchor the every-other-week pattern. Removes ambiguity without needing a calendar widget.
3. **Monthly schedule**: "Week of month" qualifier dropdown (1st, 2nd, 3rd, 4th, Last). Deterministic without a reference date.
4. **Auto-message templates**: Hardcoded in the app. Two main templates:
   - New scene welcome message (detailed onboarding — auto-posted to Discord thread, see Section 10)
   - New scene admin welcome DM (credentials + getting started — copy-paste, see Section 11)
5. **Admin approval Discord webhooks**: Scene-relevant approvals (new stores, new scenes) auto-post to the scene coordination thread AND a public #scene-updates channel. Individual resolutions handled manually via DM. See Section 3 for breakdown.
6. **New scene requests**: Surface on the Scenes tab (super admin only), not the Stores tab. Requests include suggested store names so the admin can create the scene and stores in one flow. See Section 10.
7. **Store request DM templates**: Deferred — not needed for v1.4. Admin follows up manually via Discord DM.
8. **Discord forum thread creation**: App creates new forum threads in both `#scene-requests` (on request submission) and `#scene-coordination` (on approval) via webhook API `thread_name` parameter. Preview modal lets admin review/edit the welcome message before posting to `#scene-coordination`. Thread ID stored on scene record.
9. **#scene-requests stays permanent**: Public-facing intake channel where the community can see and comment on requests. App-generated requests auto-create threads there. Manual requests from Discord users are also supported — super admin handles those manually or directs users to the in-app form for tracking.
